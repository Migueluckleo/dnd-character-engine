# Agent Identity & SDD Protocol (`.claude.md`)

## 1. Agent Identity and Role

Your primary objective is to develop and maintain a character management application by strictly applying the Spec Driven Development (SDD) methodology. You must consult this file at the beginning of every session to maintain context regarding the stack, coding standards, and architectural conventions, preventing model degradation or "Context Drift.”

## 2. SDD Workflow (Immutable)

To ensure traceability, execution must follow a strict order. You must never merge phases or write code without first documenting and validating the preceding phases.

- **Phase 1: Specify (requirements.md)** - Critical Definition Phase. Prior to any technical planning, you must generate and maintain a specification file that mandatorily contains:
  - Functional Requirements: Clear descriptions of what the system must do, strictly limited to the approved rule domain (e.g., Core 5E).
  - User Stories: Written in the "As a [role], I want [action] so that [benefit]" format.
  - Acceptance Criteria (AC): Restrictive rules and exact validations that determine if a user story is complete.
  - AI Note: It is your proactive duty to identify gaps, ambiguities, or contradictions in the user's requested rules and ask for clarification before closing requirements.md.

- **Phase 2: Plan (plan.md)**
  Based exclusively on requirements.md. Define the technology stack, relational data models, schemas, and API endpoints.

- **Phase 3: Tasks (tasks.md)**
  Decomposition of plan.md into a list of atomic, sequential, and testable tasks.

- **Phase 4: Implement**
  Execution of tasks.md one by one, writing code, validating against AC, and updating the STATUS.md file.
  - **Test-Driven Execution:** During Implementation, you must write unit tests for all pure services (e.g., Rules Engine calculations) to prove they meet the Acceptance Criteria before writing the API integration or database logic.

## 3. Universal Business Directives (Rules Engine)

The system is not a simple database manager; it is a logical engine.

- **Rule Precedence:** Specificity always overrides generality (e.g., unique entity traits overwrite standard rules).
- **Mathematical Rounding:** All divisions or fractions in system calculations must round down (floor) to the nearest whole integer, regardless of the decimal value.
- **Base Calculations:** All formulas for progression, modifiers, and scaling must be isolated in pure, testable services, calculated in real-time to avoid database inconsistencies.

## 4. Data Structure and Naming Conventions

To ensure architectural consistency across the backend, database, and frontend, the following naming rules apply immutably:

- **Code Language:** All source code (variable names, tables, columns, functions, endpoints) must be written strictly in English using `snake_case` for databases and `camelCase` for in-memory code.
- **Interface Language:** Client-facing responses and UI labels (Frontend) must project values, descriptions, and official terminology in Spanish.
- **Strict Identifiers:** All system catalogs (entity IDs, base metrics, alignments, and categories) must be stored in the database using standardized English keys (lowercase, no spaces).
- **Typing and Modeling:** Core progression entities (such as class/profession options) must be modeled to store variables like Hit Dice and proficiency identifiers to automate calculations during the Implementation Phase.

## 5. Architecture and Best Practices

- **Modularity:** The architecture must be designed to isolate domain logic into independent services, allowing for future scaling to support rule expansions without rewriting the core.
- **Strict Validation (Guardrails):** The API must reject any attempt to alter system state that violates the Acceptance Criteria defined in requirements.md using appropriate HTTP error codes.
- **Standardized Error Handling:** All API error responses must follow a consistent JSON structure. They must include an HTTP status code, a technical `developerMessage` in English, and a user-friendly `clientMessage` in Spanish.
- **State Immutability:** Pure services must never mutate the input data. They should always return new instances or calculated values to prevent unintended side effects in character state.
